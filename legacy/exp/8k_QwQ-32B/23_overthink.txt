The correct answer is **D. there is no way to express who has rights to change the access matrix itself**, and here’s why:

---

### **Explanation of the Access Matrix Approach**
The access matrix model represents security policies by defining **access rights** (e.g., read, write) between **subjects** (users/processes) and **objects** (files, devices). The matrix structure is intuitive: rows represent subjects, columns represent objects, and each cell specifies the permissions granted to a subject for an object.

---

### **Why Option D is Correct**
The critical flaw of the access matrix model is its inability to **self-protect**. Specifically:

1. **Circular Dependency Problem**:  
   The access matrix itself is an object (a "protection matrix") that defines all access rights. To modify or update the access matrix (e.g., grant/revoke permissions), some entity (e.g., an administrator) must have the right to do so.  
   - However, the access matrix cannot express this right because it would require an entry within the matrix to control its own modification.  
   - This creates a paradox: Who specifies the rights to modify the access matrix? The model lacks a mechanism to define this without introducing another layer of control (e.g., a "higher-level" access matrix), which defeats the original model's simplicity.

2. **Lack of Self-Descriptive Protection**:  
   The access matrix inherently lacks the capability to enforce its own integrity. Protection of the matrix itself requires external rules or another abstraction (e.g., a "trusted computing base" or hierarchical model), which the basic access matrix does not provide.

---

### **Why Other Options Are Incorrect**
- **A. Storage and Management Complexity**:  
  While storing the matrix for large systems can be cumbersome, this is a **practical limitation** (e.g., sparse matrices or hierarchies mitigate it), not a fundamental flaw of the model.

- **B. Inexpressiveness for Complex Requirements**:  
  The access matrix is **highly expressive**. Each cell can define any combination of rights, making it capable of representing even complex policies (e.g., role-based access, discretionary access control).

- **C. Undecidable Access Decisions**:  
  Access rights in the matrix are **explicitly defined**. Determining whether a subject has access to an object is straightforward (e.g., check the corresponding matrix entry). This is **decidable**, so this option is incorrect.

---

### **Conclusion**
The access matrix’s inability to **define who can modify the matrix itself** (Option D) is its fundamental limitation. It fails to address the critical question of **meta-authorization**—the protection of the protection mechanism. This issue necessitates additional layers of control (e.g., role-based models or capability lists) to manage the access matrix’s own permissions.
